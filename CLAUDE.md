## Commands

Tasks are orchestrated via [mise](https://mise.jdx.dev/) (see `mise.toml`):

- `mise run check` ‚Äî `cargo check` across the workspace
- `mise run test` ‚Äî `cargo test` (run a single test with `cargo test --test handlers <name>`; the integration tests live in `crates/cliview/tests/handlers.rs`)
- `mise run build-ui` ‚Äî `pnpm install && pnpm build` the SvelteKit app, then copy `frontend/build/` ‚Üí `crates/cliview/frontend/dist/` (this populates the embedded SPA)
- `mise run dev-ui` ‚Äî Vite dev server on `:5173` for iterating on the frontend against a running Rust backend
- `mise run demo` ‚Äî depends on `build-ui`; runs `cargo run -p demo-cli -- web .`

Frontend-only: `cd frontend && pnpm check` for svelte-check / TypeScript validation.

## Architecture

`cliview` is a small library that exposes three flavors of typed async Rust functions as HTTP endpoints, with a SvelteKit SPA that auto-discovers them via an introspection endpoint.

### Workspace shape

- `crates/cliview/` ‚Äî library crate (published)
- `crates/cliview/frontend/dist/` ‚Äî **checked-in** prebuilt SPA, embedded via `rust-embed` so downstream users need no Node toolchain. Regenerated by `mise run build-ui`.
- `frontend/` ‚Äî SvelteKit source (static adapter; `build/` is the output)
- `examples/demo-cli/` ‚Äî end-to-end demo exercising page/action/stream handlers

### Handler model (`crates/cliview/src/`)

The builder API in `lib.rs` (`WebApp::page` / `action` / `stream`) wraps generic async functions into type-erased trait objects stored in `Registry`:

- **Page** ‚Äî `async fn() -> Result<T: Serialize>` ‚Üí GET `/api/pages/:id`
- **Action** ‚Äî `async fn(I: Deserialize + JsonSchema) -> Result<O: Serialize>` ‚Üí POST `/api/actions/:id`. The `JsonSchema` bound is load-bearing: the schema is returned in `/api/meta` so the SPA can render an auto-form.
- **Stream** ‚Äî `async fn(StreamTx) -> Result<()>` ‚Üí GET `/api/streams/:id` as SSE. Each `tx.send(value)` emits one `data:` event; the handler runs in a spawned task and errors are logged.

`handler.rs` defines `PageFn` / `ActionFn` / `StreamFn` wrappers that erase the generic signatures into `PageHandler` / `ActionHandler` / `StreamHandler` trait objects (`async_trait` + `BoxFuture`). This is why the public builder methods have those specific generic bounds ‚Äî changing them requires matching changes to the wrappers and the trait signatures.

### Frontend serving (`server.rs` ‚Üí `assets.rs`)

`static_fallback` picks one of three sources for non-`/api/*` requests, in priority order:

1. `dev_proxy` (set via `.dev_proxy()` or `CLIVIEW_DEV` env var) ‚Äî forwards to e.g. `http://127.0.0.1:5173` for live Vite dev
2. `frontend_dir` ‚Äî serves from a disk directory (for testing a local build without rebuilding the crate)
3. Otherwise the embedded `rust-embed` bundle from `crates/cliview/frontend/dist/`

The fallback serves `index.html` for unknown paths so client-side routing works.

### Frontend (`frontend/src/`)

SvelteKit with `@sveltejs/adapter-static` (fully prerendered, `fallback: "index.html"`). `src/lib/api.ts` is the typed client: `getMeta()`, `getPage()`, `runAction()`, `subscribeStream()` (uses `EventSource`). The SPA discovers everything at runtime via `/api/meta`, so new handlers registered in Rust show up without frontend changes.

## Releasing

Publishing is driven by GitHub Actions (`.github/workflows/release.yml`) and triggered by pushing a tag:

1. Bump `version` in the root `Cargo.toml` (`[workspace.package]`) ‚Äî the crate inherits it.
2. Rebuild the embedded SPA (`mise run build-ui`) and commit the updated `crates/cliview/frontend/dist/`.
3. Tag and push: `git tag v0.1.1 && git push origin v0.1.1`.
4. The release workflow rebuilds the UI on CI, asserts the tag matches the `cliview` crate version, runs tests, and `cargo publish -p cliview` using `CRATES_IO_TOKEN` (repo secret).

Manual dry-run: trigger the `Release` workflow via `workflow_dispatch` with `dry_run=true` ‚Äî it runs `cargo publish --dry-run` without needing the token.

The crate ships with README and LICENSE-{MIT,APACHE} as symlinks from `crates/cliview/` to the repo root; `include = [...]` in `crates/cliview/Cargo.toml` enumerates what gets packaged. Verify locally with `cargo package --list -p cliview --allow-dirty`.

## Conventions

- The embedded `frontend/dist/` is a build artifact committed to the repo. After changing the Svelte code, run `mise run build-ui` before committing; `cargo` tests and the demo will otherwise use the stale embedded SPA. `crates/cliview/build.rs` re-runs when `frontend/dist` changes.
- Integration tests in `crates/cliview/tests/handlers.rs` find a free port by binding a probe, dropping it, then telling `WebApp` to `.bind()` the same address ‚Äî this is race-prone but avoids needing to plumb a listener through the public API. Poll `/api/meta` to wait for readiness.
- `anyhow::Result` is the handler-facing error type; the library converts into `Error::Handler` which maps to HTTP 500. `Error::BadRequest` (400) is produced specifically by action input deserialization failures, and `Error::NotFound` (404) by unknown handler ids.
